# CLAUDE.md — บริบทถาวรของโปรเจกต์

ไฟล์นี้ถูกอ่านอัตโนมัติทุก session ใช้เป็นกฎและความจำหลักของโปรเจกต์ ตอบและสื่อสารกับผู้ใช้ **เป็นภาษาไทยเสมอ**

---

## โปรเจกต์นี้คืออะไร

**Smart QSC Audit & Franchise Standard System** — เว็บแอป Responsive สำหรับตรวจประเมินมาตรฐานสาขาแฟรนไชส์ร้านอาหาร (กรณีศึกษา: **วิเชียรซาลาเปา · ขนมจีบ**) ครอบคลุมกระบวนการ **7 ขั้นตอน** รองรับการตรวจ 2 แบบ: **Audit ปกติ** (เปิดเผย เห็นหลังร้าน) และ **Mystery Shopper** (ลับ เห็นเฉพาะมุมลูกค้า)

เป้าหมาย: สอดคล้องเกณฑ์ **DBD พ.ศ. 2569 หมวด 6** (ตรวจแฟรนไชส์ซี ≥ 1 ครั้ง/ปี เก็บคะแนนประเมินผล)

ทั้งระบบเป็น **ภาษาไทย** วันที่เป็น **พ.ศ.** ใช้คำว่า **QSC** (ไม่ใช่ QA)

---

## แหล่งความจริงหลัก (อ่านก่อนเสมอ)

- **`docs/audit-criteria.md`** = single source of truth ของ **เกณฑ์ข้อตรวจทุกข้อ, น้ำหนัก, seed data, business rules**
- เมื่อจะแก้/เพิ่มเกณฑ์หรือ seed ต้องยึดไฟล์นี้ ถ้าจะเปลี่ยน ให้เปลี่ยนที่ไฟล์นี้ที่เดียวแล้วค่อยปรับโค้ดตาม
- ถ้าโค้ดกับไฟล์นี้ขัดกัน → ไฟล์นี้ถูกเสมอ แจ้งผู้ใช้

---

## ข้อห้ามเด็ดขาด (HARD CONSTRAINTS — ห้ามผิด)

1. **ห้ามใช้ localStorage เก็บข้อมูลการตรวจ** — ใช้ IndexedDB (Dexie.js) เท่านั้น
2. **Mystery Shopper ห้ามมีข้อที่ลูกค้ามองไม่เห็น** (ตู้แช่/สต๊อก/ถังแก๊ส/หลังครัว/บันทึกครัวกลาง)
3. **RCA แยก 3 ช่องเสมอ** (Correction / Root Cause / Preventive) ห้ามรวมช่องเดียว
4. **ผู้ติดตามผล (ขั้นตอน 6) ต้องคนละคนกับผู้ตรวจที่พบปัญหา** — บังคับ
5. **ห้ามมอบหมายผู้ตรวจที่:** มี conflict กับสาขานั้น / ยังไม่ผ่านอบรม / cert หมดอายุ / ไม่ active
6. **รายการเลือกสาขา/ผู้ตรวจ แสดงเฉพาะ Active**
7. **แผน Mystery เป็นความลับต่อแฟรนไชส์ซี** — ซีเห็นได้เฉพาะผลหลังตรวจเสร็จ
8. **วันที่ พ.ศ. + UI ภาษาไทย** ทุกที่

## 3 Roles (รวมในแอปเดียว สลับด้วยปุ่มมุมขวาบน)

- **ผู้จัดการ (HQ):** เห็นทุกอย่าง จัดการทะเบียน วางแผน มอบหมาย Dashboard รวม
- **ผู้ตรวจ (Auditor):** เห็นงานที่ได้รับมอบหมาย ตรวจ บันทึก ถ่ายรูป
- **แฟรนไชส์ซี:** เห็นเฉพาะสาขาตัวเอง ตอบ RCA ติดตามสถานะ

---

## 7 ขั้นตอน (ภาพรวม)

1. **ทะเบียนสาขา** — ซี/ซอร์, Active/Inactive (seed 20 สาขา จาก appendix)
2. **ทะเบียนผู้ตรวจ** — Internal/Independent, training gate (seed 6 คน จาก appendix)
3. **วางแผนทั้งปี** — ปกติ/Mystery, การันตี ≥1 ครั้ง/ปี, เลื่อนได้+เก็บประวัติ, กัน conflict
4. **ตรวจหน้างาน** ⭐ — สลับเกณฑ์อัตโนมัติ, conditional logic+บังคับรูป, live score, offline, PDF
5. **แจ้งผล + RCA** ⭐ — RCA 3 ช่อง, ทุกข้อ fail/warn → CAPA
6. **ติดตามผล** ⭐ — ผู้ตรวจคนใหม่ verify + after photo, ปิด CAPA ได้เมื่อ verified
7. **Dashboard** — KPI, อันดับ, เทรนด์, CAPA ค้าง, lifecycle status, รายงาน 2 มุม

**คะแนน:** ถ่วงน้ำหนักตามหมวด, N/A ไม่นับ · **เกรด:** A≥90 B≥80 C≥70 D<70 (ต่ำกว่า 80 = ต่ำกว่ามาตรฐาน)
**Lifecycle:** วางแผนแล้ว → ครบกำหนด → กำลังตรวจ → รอตอบ RCA → รอติดตามผล → ปิดรอบ

## วิธีทำงาน (WORKFLOW)

- **ทำเป็นเฟส อย่าลุยยาว** จบแต่ละเฟส → `git commit` + สรุปสั้น + ถามผู้ใช้ก่อนไปต่อ
- เฟส: 0 วางแผน(ไม่เขียนโค้ด) → 1 ฐานราก → 2 ขั้นตอน1-3 → 3 ขั้นตอน4 → 4 ขั้นตอน5-6 → 5 ขั้นตอน7 → 6 เก็บงาน
- ทุกเฟสต้อง `npm run dev` รันได้ ไม่มี console error
- ก่อนบอกว่าเสร็จ ไล่ checklist Definition of Done ให้ครบ (ดู PROMPT ข้อ 11)
