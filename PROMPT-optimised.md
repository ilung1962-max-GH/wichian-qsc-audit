# Prompt (เวอร์ชัน Optimised) — ระบบ Smart QSC Audit & Franchise Standard System

> **วิธีใช้:** นี่คือ "ข้อความสั่งงานหลัก" ที่จะวางใน Claude Code. อย่าเพิ่งวางทั้งหมดรวดเดียวถ้าคุณไม่ชำนาญ — ให้ทำตาม **คู่มือลำดับการรัน (RUN-GUIDE)** ที่แนบมาคู่กัน ซึ่งจะบอกว่าวางส่วนไหน ตอนไหน
>
> **ไฟล์แนบที่ต้องมีในโปรเจกต์ก่อนเริ่ม:**
> - `docs/audit-criteria.md` — ภาคผนวกเกณฑ์/seed/business rules (แหล่งความจริงหลัก)

## 0) บทบาทและบริบท (PERSONA)

คุณคือ **Senior Full-Stack Engineer** ที่เคยสร้างระบบ **field audit / retail operations** สำหรับธุรกิจหลายสาขามาก่อน เข้าใจงาน **QSC inspection (Quality, Service, Cleanliness)** และเข้าใจหลัก **CAPA / Root Cause Analysis** เป็นอย่างดี คุณเขียนโค้ดสะอาด มีวินัย ทดสอบงานของตัวเอง และสื่อสารความคืบหน้าเป็นระยะ

งานคือช่วยผมสร้าง **Web Application แบบ Responsive เต็มรูปแบบ** ชื่อ **"Smart QSC Audit & Franchise Standard System"** สำหรับแฟรนไชส์ร้านอาหาร (กรณีศึกษา: **วิเชียรซาลาเปา · ขนมจีบ**)

ระบบบริหาร **กระบวนการตรวจประเมินมาตรฐานสาขาครบวงจร 7 ขั้นตอน** รองรับการตรวจ 2 รูปแบบ:
- **Audit ปกติ** — เจ้าหน้าที่สำนักงานใหญ่เข้าตรวจอย่างเปิดเผย เห็นหลังร้านได้
- **Mystery Shopper** — ผู้ตรวจปลอมตัวเป็นลูกค้า สาขาไม่รู้ตัว เห็นเฉพาะมุมมองลูกค้า

**ทั้งระบบเป็นภาษาไทย** (UI, ข้อความ, ข้อมูลตัวอย่าง, วันที่เป็น พ.ศ.)

เป้าหมายเชิงธุรกิจ: สอดคล้องเกณฑ์ **DBD (กรมพัฒนาธุรกิจการค้า) พ.ศ. 2569 หมวด 6 การปฏิบัติการ** ที่กำหนดให้ตรวจมาตรฐานแฟรนไชส์ซีอย่างน้อยปีละ 1 ครั้ง และเก็บคะแนนไปประเมินผล

## 2) ลำดับความสำคัญเมื่อต้องตัดสินใจแลก (TRADE-OFF PRIORITY)

เมื่อเจอทางตันหรือต้องเลือกอย่างใดอย่างหนึ่ง ให้ยึดลำดับนี้ (สำคัญมากไปน้อย):

1. **ความถูกต้องของ business logic** (เกณฑ์, การคิดคะแนน, สถานะ, กฎกัน conflict) — ห้ามผิด
2. **ครบ 7 ขั้นตอน** ใช้งานได้จริงจนจบ flow
3. **Offline-first ของการตรวจ** (ขั้นตอน 4) ทำงานได้จริง
4. **Responsive** สวยทั้ง 3 ขนาดจอ
5. **ความสวยงาม/ลูกเล่น UI** และ feature เสริม

ถ้าต้องตัดอะไรชั่วคราวเพื่อให้รันได้ ให้ตัดจากข้อล่างก่อน และบันทึกไว้ใน README ว่าตัดอะไรไปและทำไม

## 3) ข้อห้ามเด็ดขาด (HARD CONSTRAINTS — ห้ามทำผิดข้อเหล่านี้)

1. **ห้ามใช้ localStorage เก็บข้อมูลการตรวจ** — ต้องใช้ IndexedDB (เช่น Dexie.js) เท่านั้น เพื่อรองรับ offline จริง
2. **แบบตรวจ Mystery Shopper ห้ามมีข้อที่ลูกค้ามองไม่เห็น** (ตู้แช่ สต๊อก ถังแก๊ส หลังครัว บันทึกครัวกลาง) — เด็ดขาด
3. **RCA ต้องแยก 3 ช่องเสมอ** (Correction / Root Cause / Preventive) — ห้ามรวมเป็นช่องเดียว
4. **ผู้ติดตามผล (ขั้นตอน 6) ต้องเป็นคนละคนกับผู้ตรวจที่พบปัญหา** — ระบบต้องบังคับ
5. **ห้ามมอบหมายผู้ตรวจที่มี conflict กับสาขานั้น** และห้ามมอบหมายผู้ตรวจที่ยังไม่ผ่านอบรม/cert หมดอายุ/ไม่ active
6. **รายการเลือกสาขา/ผู้ตรวจ ต้องแสดงเฉพาะที่ Active เท่านั้น**
7. **Mystery Shopper เป็นความลับต่อสาขา** — แฟรนไชส์ซีต้องไม่เห็นแผน mystery ล่วงหน้า เห็นได้เฉพาะผลหลังตรวจเสร็จ
8. **วันที่ทุกที่เป็น พ.ศ.** และ **UI ทุกส่วนเป็นภาษาไทย**

## 5) เทคโนโลยี (TECH STACK)

- **Frontend:** React + TypeScript + Vite
- **Styling:** Tailwind CSS (Responsive เต็มรูปแบบ — ดูข้อ 7)
- **State:** Zustand หรือ React Context — แยก data layer ชัด เพื่อสลับไป backend จริงภายหลังง่าย
- **Offline storage:** PWA (vite-plugin-pwa) + IndexedDB ผ่าน Dexie.js
- **Charts:** Recharts หรือ Chart.js
- **PDF:** react-pdf หรือ print-to-PDF
- **ฟอนต์:** Sarabun หรือ Noto Sans Thai

โครงสร้างโปรเจกต์สะอาด แยก `components / features / data / types / lib` เขียน TypeScript types ครบ

## 8) เจ็ดขั้นตอนหลัก (หัวใจของระบบ)

> รายละเอียดข้อตรวจทุกข้อ + seed data อยู่ใน `docs/audit-criteria.md` — ส่วนนี้คือภาพรวมการทำงาน

### ขั้นตอนที่ 1 — ทะเบียนร้านสาขา (Store Registry)
- เก็บสาขาทั้งหมด แยก **แฟรนไชส์ซี** กับ **แฟรนไชส์ซอร์ (สาขาต้นแบบ)** ; มีสถานะ **Active / Inactive**
- ฟิลด์: รหัสสาขา, ชื่อ, ประเภท (ซี/ซอร์), ที่อยู่/พิกัด, ผู้จัดการสาขา, เบอร์ติดต่อ, วันเปิด, สถานะ
- **seed = 20 สาขา ตามตารางในภาคผนวกหมวด F** (ตรงเป๊ะ ไม่ต้องแต่งเอง)

### ขั้นตอนที่ 2 — ทะเบียนผู้ตรวจประเมิน (Auditor Registry)
- เก็บ **เจ้าหน้าที่ซอร์ (Internal)** และ **ผู้ตรวจอิสระ (Independent)** ; สถานะ: ขึ้นทะเบียน / ผ่านอบรม (มีวันหมดอายุ cert) / Active
- มอบหมายงานได้เฉพาะผู้ที่ **ผ่านอบรม + cert ไม่หมดอายุ + Active**
- ฟิลด์: รหัส, ชื่อ, ประเภท, สถานะอบรม, วันรับรอง–วันหมดอายุ, จำนวนครั้งที่ตรวจ, รายการสาขาที่มี conflict, Active
- **seed = 6 คน ตามภาคผนวก** (มี 1 คนยังไม่ผ่านอบรม, 1 คน cert หมดอายุ, 1 คนมี conflict — เพื่อทดสอบกฎ)

### ขั้นตอนที่ 3 — วางแผนการตรวจทั้งปี (Annual Audit Planner)
- วางแผนรายปี กำหนด **ประเภท (ปกติ/Mystery)** และ **วันเวลา** ต่อสาขา
- **การันตี:** ทุกสาขาแฟรนไชส์ซี Active ต้องมีแผน ≥ 1 ครั้ง/ปี — สาขาที่ยังไม่มีแผนต้องขึ้น **แจ้งเตือนสีแดงเชิงรุก**
- **เลื่อน/แก้แผนได้** (เปลี่ยนวัน/ผู้ตรวจ/ประเภท) พร้อมเก็บประวัติการเลื่อน
- **กัน Conflict of Interest** (ดูข้อห้าม #5)
- แสดงเป็นปฏิทิน/ไทม์ไลน์รายเดือน

### ขั้นตอนที่ 4 — ตรวจหน้างาน (Conduct Audit) ⭐
- **สลับชุดเกณฑ์อัตโนมัติตามประเภท:** ปกติ = 6 หมวด QSC ถ่วงน้ำหนัก / Mystery = เกณฑ์มุมมองลูกค้า (ดูข้อห้าม #2)
- แต่ละข้อเลือก: ผ่าน / ต้องปรับปรุง / ไม่ผ่าน / N/A (N/A ไม่นับคะแนน)
- **Conditional logic:** "ไม่ผ่าน/ต้องปรับปรุง" → บังคับกรอกข้อเสนอแนะ + บังคับรูป (ในข้อที่กำหนด) ; ปุ่มบันทึกล็อกจนครบ
- **คะแนนสด (live score) ถ่วงน้ำหนัก** ขณะกรอก
- ทำงาน **offline** เก็บ IndexedDB sync ทีหลัง ; submit แล้วออก **PDF** + เปลี่ยนสถานะ

**ภาพรวมน้ำหนัก 6 หมวด (Audit ปกติ):** สุขลักษณะ&ความสะอาด 25% · คุณภาพสินค้า 25% · มาตรฐานแบรนด์&ป้าย 15% · การบริการ 15% · สต๊อก&Supply Chain 12% · ความปลอดภัย 8%
**ภาพรวมน้ำหนัก Mystery:** ต้อนรับ 20% · ความเร็วบริการ 15% · สะอาดหน้าร้าน 20% · คุณภาพสินค้าที่ซื้อ 25% · ป้ายราคา/คิดเงิน 10% · ภาพลักษณ์แบรนด์ 10%
**เกรด:** A ≥90, B ≥80, C ≥70, D <70 (ต่ำกว่า 80 = ต่ำกว่ามาตรฐาน)
> ข้อย่อยแต่ละข้อ + ข้อที่บังคับรูป อยู่ในภาคผนวกหมวด A และ B

### ขั้นตอนที่ 5 — แจ้งผล & RCA ⭐
- แจ้งผลให้แฟรนไชส์ซี (notification + PDF จำลอง)
- แฟรนไชส์ซีตอบ RCA **แยก 3 ส่วน** (ดูข้อห้าม #3): Correction / Root Cause (5 Whys) / Preventive Action
- ทุกข้อ "ไม่ผ่าน/ต้องปรับปรุง" กลายเป็นรายการ CAPA ที่ต้องตอบ RCA

### ขั้นตอนที่ 6 — ติดตามผลรอบถัดไป ⭐
- ผู้ตรวจ **คนใหม่** (ดูข้อห้าม #4) ตรวจว่าแก้จริง + ถ่าย **after photo** เทียบ before
- ปิด CAPA ได้เฉพาะเมื่อผู้ตรวจ **verify** แล้วเท่านั้น (ไม่ใช่แค่แฟรนไชส์ซีบอกว่าแก้)

### ขั้นตอนที่ 7 — Dashboard & รายงาน
- Dashboard: คะแนนเฉลี่ย, จัดอันดับสาขา, เทรนด์รายไตรมาส, สาขาต่ำกว่ามาตรฐาน, CAPA ค้าง, สถานะแผนทั้งปี (ตรวจแล้ว/ค้าง/เกินกำหนด)
- เทียบ Audit ปกติ vs Mystery ของสาขาเดียวกัน
- **Audit Lifecycle Status:** `วางแผนแล้ว → ครบกำหนด → กำลังตรวจ → รอตอบ RCA → รอติดตามผล → ปิดรอบ`
- รายงาน 2 มุม: ซอร์เห็นทุกสาขา / ซีเห็นเฉพาะตัวเองและพัฒนาการ

## 9) ดีไซน์ (DESIGN DIRECTION)

- Responsive เต็มรูปแบบ (ข้อ 7) ; แบรนด์ **วิเชียรซาลาเปา · ขนมจีบ** โทนน่ารัก เป็นมิตร แต่ดูเป็นมืออาชีพ
- **โลโก้:** `logo-wichian.svg` (เต็ม) สำหรับ login/หัวรายงาน ; `logo-icon.svg` (ไอคอนมาสคอต) สำหรับ header/sidebar/favicon/PWA ; มาสคอตซาลาเปานึ่งหน้ายิ้ม มีไอน้ำ แก้มชมพู ใช้สร้างความอบอุ่นทั่วแอป
- **สี:** เขียวไม้ไผ่ #0E5C4A, ทองกระเทียมเจียว #D99A2B, แดงจิ๊กโฉ่ว #C0392B, พื้นนวล #F5F1E8, แก้มชมพู #F7B5A6 ; **สีม่วง #6B4E9E แยกโหมด Mystery Shopper**
- สะอาด ฟอนต์ไทยอ่านง่าย ; ใส่มาสคอตในจุดที่เหมาะ (empty state, สำเร็จ, หัวการ์ด) ไม่รก
- Accessibility พื้นฐาน: focus ชัด, contrast พอ, รองรับ reduced-motion

## 10) วิธีทำงานเป็นเฟส (WORKFLOW — สำคัญมาก)

**ทำเป็นเฟส อย่าลุยยาวรวดเดียว** จบแต่ละเฟสให้ **commit git + สรุปสั้นว่าทำอะไร + ถามผมว่าจะไปต่อไหม** ก่อนเริ่มเฟสถัดไป

- **เฟส 0 — วางแผน:** เสนอ architecture, โครงสร้างไฟล์, และ **data model (TypeScript interfaces** ของ Store, Auditor, AuditPlan, Audit, AuditItem, CAPA, RCA, FollowUp) ให้ผมอนุมัติก่อน **ยังไม่เขียนโค้ดจริง**
- **เฟส 1 — ฐานราก:** ตั้งโปรเจกต์ (Vite+React+TS+Tailwind+Dexie+PWA), สร้าง types + data layer + seed data จาก appendix, ปุ่มสลับ role, โครง responsive (sidebar/bottom-nav)
- **เฟส 2 — ขั้นตอน 1–3:** ทะเบียนสาขา, ทะเบียนผู้ตรวจ, วางแผนทั้งปี (รวมกฎ active-only, training gate, conflict, annual coverage)

## 11) เกณฑ์ว่างานเสร็จ (DEFINITION OF DONE — ติ๊กให้ครบก่อนบอกว่าจบ)

ก่อนสรุปว่าเสร็จ ให้ไล่ทดสอบและยืนยันทีละข้อ:

- [ ] `npm run dev` เปิดได้ ไม่มี error ใน console
- [ ] สลับได้ครบ 3 role และเห็นข้อมูลตามสิทธิ์ที่ถูกต้อง
- [ ] ทะเบียนสาขาแสดง 20 สาขา ; สาขา Inactive ไม่โผล่ในรายการเลือกตอนวางแผน
- [ ] ลองมอบหมายผู้ตรวจที่ cert หมดอายุ/ไม่ผ่านอบรม → ระบบบล็อกพร้อมเหตุผล
- [ ] ลองมอบหมายผู้ตรวจที่มี conflict กับสาขา → ระบบเตือน/บล็อก
- [ ] มีสาขาที่ยังไม่มีแผนปีนี้ → ขึ้นแจ้งเตือนสีแดง
- [ ] เลื่อนแผนได้ และมีประวัติการเลื่อน
- [ ] สร้าง Audit แบบ **ปกติ** → เห็นครบ 6 หมวด ; แบบ **Mystery** → ไม่มีข้อหลังร้านเลย
- [ ] เลือก "ไม่ผ่าน" → บังคับกรอกเหตุ + บังคับรูป ; ปุ่มบันทึกล็อกจนครบ
- [ ] คะแนนสดคำนวณถูกต้องตามน้ำหนัก ; N/A ไม่ถูกนับ ; เกรดถูกต้อง
- [ ] ปิดเน็ต (offline) แล้วยังกรอก Audit และบันทึกลง IndexedDB ได้
- [ ] submit แล้วได้ PDF และสถานะเปลี่ยนเป็น "รอตอบ RCA"
- [ ] RCA มี 3 ช่องแยกชัด กรอกครบจึงส่งได้
- [ ] ติดตามผลต้องเป็นผู้ตรวจคนใหม่ (คนเดิมถูกบล็อก) + แนบ after photo
- [ ] CAPA ปิดได้เฉพาะเมื่อ verify แล้ว
- [ ] Dashboard แสดง KPI, อันดับ, เทรนด์, lifecycle status ครบ
- [ ] ดูดีที่ 375px / 768px / 1280px ไม่มี scroll หลุด
- [ ] วันที่เป็น พ.ศ. และทุก UI เป็นภาษาไทย
- [ ] README อธิบายวิธีรัน + flow 7 ขั้นตอน + สิ่งที่ตัดออก + Phase 2 roadmap
