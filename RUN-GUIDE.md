# คู่มือลำดับการรันบน Claude Code (RUN-GUIDE) — สำหรับผู้เริ่มต้น

> คู่มือนี้ออกแบบสำหรับคนที่ **ไม่เคยใช้ Claude Code มาก่อน** ทำตามทีละขั้น ไม่ต้องเดา. เป้าหมายคือสร้างระบบ Smart QSC Audit ให้สำเร็จอย่างถูกต้องและปลอดภัยที่สุด
>
> **หลักการสำคัญที่สุด:** อย่ารีบ. ให้ Claude Code ทำ **ทีละเฟส** แล้วหยุดให้เราดูก่อนไปต่อ. ถ้างงตรงไหน ให้พิมพ์ถามมันเป็นภาษาไทยได้เลย — มันตอบและอธิบายได้

---

## ส่วน A — เตรียมเครื่อง (ทำครั้งเดียว)

ถ้าคุณยังไม่มี Claude Code ในเครื่อง ทำตามนี้ก่อน:

1. **ติดตั้ง Node.js** (เวอร์ชัน 18 ขึ้นไป)
   - ไปที่ https://nodejs.org โหลดตัว "LTS" มาติดตั้ง (กด Next ไปเรื่อยๆ)
   - ตรวจว่าติดตั้งสำเร็จ: เปิดโปรแกรม **Terminal** (Mac) หรือ **PowerShell** (Windows) พิมพ์:
     ```
     node -v
     ```
     ถ้าขึ้นตัวเลขเช่น `v20.x.x` แปลว่าใช้ได้

2. **ติดตั้ง Claude Code**
   - ในหน้าต่างเดิม พิมพ์:
     ```
     npm install -g @anthropic-ai/claude-code
     ```
   - รอจนเสร็จ

3. **เตรียมโฟลเดอร์โปรเจกต์**
   - สร้างโฟลเดอร์ใหม่สำหรับงานนี้ (เช่นชื่อ `wichian-audit`) ไว้บนหน้า Desktop ก็ได้
   - ในโฟลเดอร์นั้น สร้างโฟลเดอร์ย่อยชื่อ `docs` และ `public`
   - วางไฟล์ให้ครบ (ชื่อไฟล์ถูกต้องพร้อมใช้แล้ว แค่วางให้ถูกโฟลเดอร์):
     - `CLAUDE.md`  ← **วางในโฟลเดอร์หลัก** (Claude Code อ่านอัตโนมัติทุกครั้ง เป็นความจำของโปรเจกต์)
     - `docs/audit-criteria.md`  ← ไฟล์ภาคผนวกเกณฑ์ (วางในโฟลเดอร์ `docs`)
     - `public/logo-wichian.svg`  ← โลโก้เต็ม
     - `public/logo-icon.svg`  ← ไอคอน
     - เก็บไฟล์ `PROMPT-optimised.md` ไว้ในโฟลเดอร์หลักด้วย (ไว้คัดลอกข้อความ)

   โครงสร้างควรเป็นแบบนี้:
   ```
   wichian-audit/
   ├── CLAUDE.md            ← ความจำถาวร (Claude Code อ่านเองทุกครั้ง)
   ├── docs/
   │   └── audit-criteria.md
   ├── public/
   │   ├── logo-wichian.svg
   │   └── logo-icon.svg
   └── PROMPT-optimised.md
   ```

## ส่วน B — เปิด Claude Code

1. เปิด Terminal/PowerShell
2. เข้าไปในโฟลเดอร์โปรเจกต์ ด้วยคำสั่ง `cd` ตามด้วยที่อยู่โฟลเดอร์ เช่น:
   ```
   cd Desktop/wichian-audit
   ```
3. พิมพ์เปิด Claude Code:
   ```
   claude
   ```
4. ครั้งแรกมันจะให้ login ตามขั้นตอนบนหน้าจอ (ใช้บัญชี Claude ของคุณ) — ทำตามที่มันบอก

> **เคล็ดลับ:** เมื่ออยู่ในหน้าจอ Claude Code คุณพิมพ์สั่งงานเป็น **ภาษาไทยได้เลย** ทุกอย่าง

## ส่วน C — ลำดับการรัน 7 ก้าว (ทำตามนี้เป๊ะ)

### ก้าวที่ 1 — ให้มันอ่านไฟล์และวางแผนก่อน (ยังไม่ให้เขียนโค้ด)

คัดลอก **เนื้อหาทั้งหมด** ในไฟล์ `PROMPT-optimised.md` วางลงไปในช่องพิมพ์ของ Claude Code แล้วกด Enter

> **หมายเหตุเรื่อง CLAUDE.md:** ไฟล์ `CLAUDE.md` ที่วางไว้ในโฟลเดอร์หลัก Claude Code จะอ่านเองอัตโนมัติทุกครั้งที่เปิด — เป็น "ความจำถาวร" ที่ทำให้มันจำกฎสำคัญของโปรเจกต์ได้แม้เปิดใหม่คนละวัน คุณไม่ต้องคัดลอกมันไปวาง แค่มีไฟล์อยู่ในโฟลเดอร์ก็พอ

> Prompt นี้สั่งให้มันเริ่มจาก "เฟส 0" คือ อ่านไฟล์เกณฑ์ + เสนอแผนโครงสร้างและ data model ให้เราดูก่อน **โดยยังไม่เขียนโค้ดจริง** — นี่เป็นจุดสำคัญที่ทำให้คนไม่เก่งก็คุมงานได้

**สิ่งที่จะเกิดขึ้น:** มันอาจถามกลับ 1–3 คำถาม กับเสนอแผน architecture + รายการ data model

**สิ่งที่คุณต้องทำ:**
- ตอบคำถามมันสั้นๆ (ถ้าไม่แน่ใจ พิมพ์ว่า "ใช้ตามที่คุณแนะนำได้เลย")
- ดูแผนคร่าวๆ ว่าครบ 7 ขั้นตอนไหม ถ้าโอเค พิมพ์ว่า:
  ```
  อนุมัติแผน เริ่มเฟส 1 ได้เลย
  ```

---

### ก้าวที่ 2 — เฟส 1: วางฐานราก

หลังอนุมัติ มันจะตั้งโปรเจกต์ + สร้าง data layer + seed data + โครง responsive

**รอจนมันบอกว่าเฟส 1 เสร็จ** แล้วลองเปิดดูของจริง พิมพ์:
```
รัน npm run dev ให้หน่อย แล้วบอก URL ที่เปิดดูได้
```
มันจะให้ลิงก์ (มักเป็น `http://localhost:5173`) — เปิดในเบราว์เซอร์เพื่อดู

ถ้าเปิดแล้วเห็นหน้าตั้งต้น (มีโลโก้/ปุ่มสลับ role) แปลว่าฐานรากโอเค พิมพ์:
```
ดีแล้ว commit งานเฟสนี้ แล้วไปเฟส 2
```

> **ถ้าเจอ error:** คัดลอกข้อความ error ทั้งหมด วางให้มัน แล้วพิมพ์ว่า "แก้ error นี้ให้หน่อย" — มันจะแก้ให้

---

### ก้าวที่ 3 — เฟส 2: ทะเบียนสาขา + ผู้ตรวจ + วางแผน (ขั้นตอน 1–3)

มันจะสร้าง 3 หน้าแรก พอเสร็จ ให้ทดสอบกฎสำคัญ พิมพ์ทีละคำสั่ง:
```
รัน npm run dev แล้วบอก URL
```
เปิดดู แล้วลองกดเล่นในเบราว์เซอร์:
- เข้าหน้าวางแผน → ลองมอบหมายผู้ตรวจที่ cert หมดอายุ → **ต้องถูกบล็อก**
- ดูว่ามีสาขาที่ขึ้นเตือนสีแดง "ยังไม่มีแผนปีนี้" ไหม

ถ้าครบ พิมพ์:
```
ทดสอบผ่าน commit แล้วไปเฟส 3
```

---

### ก้าวที่ 4 — เฟส 3: หน้าตรวจร้าน (ขั้นตอน 4 — หัวใจ)

นี่คือส่วนสำคัญที่สุด พอมันบอกเสร็จ เปิดดูแล้วลองในเบราว์เซอร์:
- เริ่มตรวจแบบ **ปกติ** → ต้องเห็นครบ 6 หมวด
- เริ่มตรวจแบบ **Mystery** → ต้อง **ไม่มี** ข้อเรื่องตู้แช่/สต๊อก/ถังแก๊ส
- ลองเลือก "ไม่ผ่าน" สักข้อ → ต้องบังคับให้กรอกเหตุผล + แนบรูป
- ลองปิด WiFi/เน็ต แล้วกรอกต่อ → **ต้องยังกรอกและเซฟได้**

ถ้าครบ:
```
ทดสอบผ่าน commit แล้วไปเฟส 4
```

> **วิธีทดสอบ offline ง่ายๆ:** ในเบราว์เซอร์ Chrome กด F12 → แท็บ Network → เปลี่ยน dropdown เป็น "Offline" แล้วลองกรอกฟอร์มต่อ

---

### ก้าวที่ 5 — เฟส 4: แจ้งผล + RCA + ติดตามผล (ขั้นตอน 5–6)

เปิดดูแล้วลอง:
- สลับเป็น role **แฟรนไชส์ซี** → ตอบ RCA → ต้องมี **3 ช่องแยก** (แก้เฉพาะหน้า / รากปัญหา / กันซ้ำ)
- ไปหน้าติดตามผล → ลองให้ผู้ตรวจคนเดิม follow-up → **ต้องถูกบล็อก** (ต้องเป็นคนใหม่)

ถ้าครบ:
```
ทดสอบผ่าน commit แล้วไปเฟส 5
```

---

### ก้าวที่ 6 — เฟส 5: Dashboard (ขั้นตอน 7)

เปิดดู Dashboard ว่ามีครบ: คะแนนเฉลี่ย, อันดับสาขา, เทรนด์, CAPA ค้าง, สถานะแผนทั้งปี
ลองย่อ-ขยายหน้าต่างเบราว์เซอร์ดูว่า responsive ไหม

ถ้าครบ:
```
commit แล้วไปเฟส 6 เก็บงาน
```

---

### ก้าวที่ 7 — เฟส 6: ตรวจรับงานขั้นสุดท้าย

พิมพ์สั่งให้มันตรวจตัวเองตาม checklist:
```
ไล่ตรวจ Definition of Done ในข้อ 11 ของ Prompt ทีละข้อ บอกผมว่าข้อไหนผ่าน ข้อไหนยัง แล้วแก้ให้ครบ
```
มันจะไล่เช็คและแก้จุดที่ตกหล่น พอครบให้พิมพ์:
```
เขียน README ให้ครบ แล้ว commit งานทั้งหมดเป็นครั้งสุดท้าย
```

**เสร็จแล้ว!** ระบบพร้อมใช้งาน demo — เปิดดูด้วย `npm run dev` ได้ตลอด

## ส่วน D — คำสั่งที่ใช้บ่อย (จำไว้ใช้ได้ทุกเมื่อ)

| อยากทำอะไร | พิมพ์ใน Claude Code |
|---|---|
| เปิดดูแอป | `รัน npm run dev แล้วบอก URL` |
| เจอ error | วาง error ทั้งหมด แล้วพิมพ์ `แก้ error นี้ให้หน่อย` |
| ไม่เข้าใจสิ่งที่มันทำ | `อธิบายเป็นภาษาไทยง่ายๆ ว่าเพิ่งทำอะไรไป` |
| อยากให้บันทึกงาน | `commit งานตอนนี้ พร้อมข้อความสรุปสั้นๆ` |
| งานเพี้ยนจากที่ขอ | `อันนี้ไม่ตรงกับ Prompt ข้อ X ช่วยแก้ให้ตรง` |
| อยากย้อนกลับ | `ขอ undo การแก้ล่าสุด` หรือ `ย้อนกลับไป commit ก่อนหน้า` |
| หยุดพักไว้ก่อน | ปิดหน้าต่างได้เลย งานที่ commit แล้วไม่หาย ครั้งหน้าพิมพ์ `claude` ในโฟลเดอร์เดิมเพื่อทำต่อ |

## ส่วน E — กฎเหล็ก 5 ข้อสำหรับมือใหม่

1. **ทีละเฟส อย่ารีบ** — ให้มันหยุดให้เราดูก่อนทุกเฟส อย่าพิมพ์ "ทำทั้งหมดเลย"
2. **เปิดดูของจริงทุกเฟส** — อย่าเชื่อแค่ที่มันพิมพ์ว่าเสร็จ ให้เปิดเบราว์เซอร์ดูเอง
3. **commit บ่อยๆ** — ทุกเฟสที่ผ่าน สั่ง commit ไว้ จะได้ย้อนกลับได้ถ้าพัง
4. **error ไม่ใช่เรื่องน่ากลัว** — แค่คัดลอกไปให้มันแก้ มันแก้ได้เกือบทุกอย่าง
5. **พิมพ์ถามเป็นไทยได้เสมอ** — ไม่เข้าใจอะไรถามได้ ไม่ต้องเกรงใจ

## ส่วน F — ถ้าอยากเอาขึ้นใช้งานจริง (ภายหลัง)

ตอน demo เสร็จแล้ว ถ้าจะใช้จริงกับหลายสาขา ค่อยกลับมาบอกมันทีหลังว่า:
```
ช่วยต่อ backend จริงด้วย Supabase (auth + database + storage รูปภาพ) ตาม Phase 2 roadmap ใน README
```
Supabase เหมาะที่สุดเพราะได้ระบบ login, ฐานข้อมูล sync, และที่เก็บรูป ครบในที่เดียว ลงทุนเริ่มต้นต่ำ — แต่ส่วนนี้ค่อยทำตอนพร้อม ไม่ต้องรีบในรอบแรก
